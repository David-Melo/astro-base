# Generate Overview

Use this skill to generate or regenerate the project's documentation summaries.

## What Gets Generated

When invoked, generate these files:

| File | Purpose | Audience |
|------|---------|----------|
| `docs/project_overview.md` | Technical stack and project structure summary | AI agents and internal contributors |
| `docs/site_overview.md` | Implementation summary and handoff snapshot | Stakeholders, handoff, internal contributors |
| `README.md` | Developer documentation | Contributors |

Do not generate overview files into `.claude/`.

## Sources Of Truth

`docs/strategy_blueprint.md` and `docs/brand_palette.md` remain the primary authority for brand, messaging, audience, and site structure. The files generated by this workflow are summaries and references. They must not override strategy.

### Client-Specific

- `docs/strategy_blueprint.md`
- `docs/brand_palette.md`
- `docs/sitemap.md`
- `docs/design-brief.md` when present

### Project-Specific

- `package.json`
- `src/components/`
- `src/pages/`
- `src/content/`
- `src/styles/`
- `public/`
- `astro.config.mjs`
- `tsconfig.json`
- `postcss.config.cjs`

## Workflow

### 1. Gather Information

- extract client identity, audience, GEO, tone, and differentiators from `docs/strategy_blueprint.md`
- extract color system and aesthetic direction from `docs/brand_palette.md`
- scan the codebase for components, pages, content files, styles, fonts, and assets
- read dependencies and configuration files for the real technical stack

### 2. Capture Future-Agent Context Explicitly

When building `docs/project_overview.md`, make sure future agents can answer questions like:

- what framework is this site built with?
- what styling system is in use?
- what UI/component library is available?
- what content system or content collections exist?
- which package dependencies are available to use?
- what path aliases and config conventions exist?
- what scripts should be used for development, build, and project workflows?
- what major integrations, generators, or helper systems already exist?

Do not leave these implied. State them directly.

### 3. Generate `docs/project_overview.md`

Create a technical project summary for agents and contributors.

Include:

- project type
- tech stack
- framework and version
- styling system and version
- UI library and component approach
- content system and content locations
- available npm/package dependencies worth knowing about
- Astro/Vite/PostCSS integrations and plugins
- project structure
- TypeScript configuration
- path aliases
- build and development notes
- project scripts
- typography and styling notes
- current implementation state
- project-specific constraints and conventions

Use concrete names whenever possible, for example:

- `Astro`
- `Tailwind CSS v4`
- `Preline UI`
- `MDX`
- `astro:content`

If a dependency or integration materially affects implementation decisions, mention it explicitly rather than burying it in a generic list.

### 4. Suggested `docs/project_overview.md` Section Shape

Prefer a structure that makes the environment easy to scan:

- Project Type
- Core Stack
- Key Dependencies
- Config And Integrations
- Project Structure
- Content Architecture
- Component Architecture
- Styling System
- Scripts And Commands
- Current State
- Constraints And Conventions

### 5. Generate `docs/site_overview.md`

Create the implementation summary using the same output contract as the site overview workflow.

Include:

- project status
- brand implementation
- components created
- pages and content
- key features
- technical stack
- responsive breakpoints
- next steps
- build status
- deployment notes
- contact information

### 6. Generate `README.md`

Create developer-facing documentation for contributors.

Include:

- project summary
- tech stack
- getting started
- project structure
- scripts
- deployment
- documentation links

## Output Policy

- generated overview summaries belong in `docs/`
- the root `README.md` is the only root-level document produced by this workflow
- do not generate `.claude/rules/overview.md`
- do not overwrite the base root `CLAUDE.md` entrypoint
- do not generate root `SITE_OVERVIEW.md`

## Quick Usage

- generate all project docs
- regenerate docs after major implementation changes
- create the initial docs set for a new project with strategy already in place

## Validation Checklist

- `docs/project_overview.md` reflects the real codebase structure and stack
- `docs/project_overview.md` explicitly states the framework, styling system, UI library, content system, major dependencies, integrations, path aliases, and scripts
- `docs/site_overview.md` accurately lists implemented components, routes, and current status
- `README.md` is contributor-friendly and aligned with the real project scripts
- none of the generated summaries contradict `docs/strategy_blueprint.md`
